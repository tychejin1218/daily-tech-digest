### 커버링 인덱스(Covering Index)로 쿼리 성능 극대화

커버링 인덱스는 쿼리에서 필요한 모든 컬럼이 인덱스 자체에 포함되어, 실제 테이블 데이터(힙)에 접근하지 않고 인덱스만으로 결과를 반환하는 기법이다. 예를 들어 `SELECT user_id, email FROM users WHERE status = 'active'`라면 `(status, user_id, email)`로 복합 인덱스를 구성하면 테이블 랜덤 I/O가 완전히 제거된다. MySQL의 `EXPLAIN` 결과에서 `Extra: Using index`가 표시되면 커버링 인덱스가 적용된 것이다.

> 💡 **왜 중요한가**: 대용량 테이블에서 랜덤 I/O 제거는 디스크 접근 비용을 수십 배 줄여 응답 시간을 극적으로 단축시킨다.

### JPA N+1 문제와 Fetch Join vs. Batch Size 선택 기준

N+1 문제는 연관 엔티티를 Lazy 로딩할 때 루트 쿼리 1번 + 각 행마다 추가 쿼리 N번이 발생하는 현상이다. `@EntityGraph`나 JPQL의 `JOIN FETCH`는 카르테시안 곱 위험이 있어 컬렉션이 여러 개일 때 `MultipleBagFetchException`이 발생하므로, 이 경우 `@BatchSize` 또는 `spring.jpa.properties.hibernate.default_batch_fetch_size`를 설정해 IN 절로 묶어 조회하는 방식이 더 안전하다. 단일 컬렉션이면 Fetch Join, 다중 컬렉션이면 Batch Size가 일반적인 선택 기준이다.

> 💡 **왜 중요한가**: N+1 문제는 개발 환경에서 잘 드러나지 않지만 프로덕션 데이터에서 수백~수천 개의 불필요한 쿼리로 DB를 포화시킬 수 있다.
