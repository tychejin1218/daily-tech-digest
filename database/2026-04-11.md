### 파티셔닝(Partitioning) vs 샤딩(Sharding) — 대용량 테이블 분산 전략

파티셔닝은 **하나의 DB 서버 안**에서 테이블을 물리적으로 분할하는 기법으로, Range·List·Hash 방식으로 데이터를 나눠 쿼리 시 불필요한 파티션을 스캔하지 않는 **파티션 프루닝(Partition Pruning)** 효과를 얻는다. 샤딩은 데이터를 **여러 DB 서버**에 수평 분산하는 방식으로 단순 파티셔닝으로는 해결되지 않는 쓰기 병목이나 수백억 건 규모에 사용하며, 크로스-샤드 조인·트랜잭션이 불가능해진다는 트레이드오프가 따른다. 실무에서는 "파티셔닝으로 버틸 수 있는 규모인가"를 먼저 확인하고, 불가피할 때만 샤딩으로 넘어가는 것이 복잡도 관리에 유리하다.

```sql
-- MySQL Range Partitioning 예시
CREATE TABLE orders (
    id       BIGINT,
    order_dt DATE,
    amount   DECIMAL(10,2)
)
PARTITION BY RANGE (YEAR(order_dt)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026)
);
-- WHERE order_dt BETWEEN '2025-01-01' AND '2025-12-31' → p2025만 스캔
```

> 💡 **왜 중요한가**: 파티셔닝과 샤딩의 경계를 혼동하면 불필요하게 어플리케이션 레이어 복잡도를 높이거나, 반대로 너무 늦게 샤딩을 도입해 서비스 장애로 이어질 수 있다.

### 인덱스 선택도(Selectivity)와 카디널리티(Cardinality) — 인덱스가 실제로 타지는 기준

카디널리티는 컬럼의 **고유 값 개수**, 선택도는 `고유 값 수 / 전체 행 수`로 0~1 사이 값이다. 옵티마이저는 선택도가 낮은(예: `is_deleted` 같은 boolean 컬럼) 인덱스를 풀 스캔보다 느리다고 판단해 **무시**하는 경우가 많다. 반대로 선택도가 높은 컬럼(UUID, 이메일 등)은 인덱스 효과가 극대화되므로, 인덱스를 설계할 때 "이 컬럼으로 걸러지는 비율이 충분한가"를 먼저 수치로 확인해야 한다.

```sql
-- 선택도 직접 계산
SELECT
    COUNT(DISTINCT status)   AS cardinality,
    COUNT(*)                 AS total_rows,
    COUNT(DISTINCT status) / COUNT(*) AS selectivity  -- 낮을수록 인덱스 효과 미미
FROM orders;

-- 실행 계획으로 옵티마이저 판단 확인
EXPLAIN SELECT * FROM orders WHERE status = 'PENDING';
-- rows 수치가 전체 테이블의 20% 이상이면 인덱스를 타더라도 오히려 손해일 수 있음
```

> 💡 **왜 중요한가**: 인덱스를 만들었는데 왜 안 타는지 모르는 대부분의 케이스는 선택도 문제이며, 이를 알면 "인덱스 추가 → 성능 개선"이라는 잘못된 공식에서 벗어날 수 있다.
