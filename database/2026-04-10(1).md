### 데드락(Deadlock) 탐지와 예방 전략

두 트랜잭션이 서로 상대방이 점유한 락을 기다리며 영원히 대기하는 상태를 데드락이라 한다. DB 엔진은 주기적으로 대기 그래프(wait-for graph)의 사이클을 탐지해 한쪽 트랜잭션을 롤백하지만, 예방이 훨씬 비용이 낮다. 핵심 예방 원칙은 **항상 동일한 순서로 테이블/행에 락을 획득**하는 것이며, 트랜잭션을 짧게 유지하고 불필요한 `SELECT FOR UPDATE`를 제거하는 것도 필수다.

```sql
-- 데드락 유발 패턴 (두 세션이 반대 순서로 락 획득)
-- Session A: UPDATE orders → UPDATE payments
-- Session B: UPDATE payments → UPDATE orders

-- 예방: 항상 orders → payments 순서로 통일
BEGIN;
  UPDATE orders  SET status = 'processing' WHERE id = 1;
  UPDATE payments SET status = 'captured'  WHERE order_id = 1;
COMMIT;

-- MySQL: 최근 데드락 원인 확인
SHOW ENGINE INNODB STATUS;  -- LATEST DETECTED DEADLOCK 섹션

-- PostgreSQL: 데드락 로그 활성화
-- postgresql.conf
-- log_lock_waits = on
-- deadlock_timeout = 1s
```

> 💡 **왜 중요한가**: 데드락은 무작위처럼 보이지만 락 획득 순서를 코드베이스 전체에서 일관되게 강제하면 구조적으로 제거할 수 있다.

### B-Tree vs Hash Index — 자료구조에 따른 인덱스 선택

B-Tree 인덱스는 정렬된 트리 구조로 **범위 검색(`BETWEEN`, `>`, `<`, `LIKE 'prefix%'`)과 정렬(`ORDER BY`)** 모두 지원하며 대부분의 상황에 적합하다. Hash 인덱스는 키를 해시 버킷에 매핑해 **동등 비교(`=`)에서 O(1)** 에 가까운 속도를 내지만, 범위 검색·정렬이 불가능하고 MySQL InnoDB는 어댑티브 해시 인덱스를 내부적으로만 사용한다. PostgreSQL은 `USING HASH`로 명시적 Hash 인덱스를 생성할 수 있으나, 실무에서는 B-Tree가 범용성에서 압도적으로 유리하다.

```sql
-- PostgreSQL: Hash 인덱스 (동등 비교 전용)
CREATE INDEX idx_session_token_hash
    ON sessions USING HASH (token);

-- B-Tree (기본값): 범위·정렬·LIKE 접두어 모두 지원
CREATE INDEX idx_orders_created_at
    ON orders (created_at);  -- USING BTREE 생략 가능

-- Hash 인덱스가 범위 검색을 지원하지 않는 예
-- 아래 쿼리는 Hash 인덱스를 사용할 수 없어 Seq Scan으로 폴백됨
SELECT * FROM orders WHERE created_at > '2026-01-01';

-- 인덱스 타입 확인 (PostgreSQL)
SELECT indexname, indexdef
FROM   pg_indexes
WHERE  tablename = 'sessions';
```

> 💡 **왜 중요한가**: 인덱스 자료구조를 이해하면 "왜 이 인덱스가 사용되지 않는가"를 실행 계획 없이도 직관적으로 예측할 수 있다.
