### 1. 커서 기반 페이지네이션(Cursor-based Pagination) — OFFSET의 성능 함정 피하기

일반적인 `OFFSET ... LIMIT` 방식은 페이지가 뒤로 갈수록 건너뛸 행을 모두 스캔해야 하므로 O(N) 비용이 발생한다. 커서 기반 페이지네이션은 이전 페이지의 마지막 행 식별자(예: `WHERE id > :last_seen_id ORDER BY id LIMIT 20`)를 조건으로 사용하여 인덱스 탐색만으로 다음 페이지를 가져온다. 수백만 건의 테이블에서 1000페이지 이후 조회 시 OFFSET은 수 초가 걸리지만, 커서 방식은 첫 페이지와 동일한 응답 속도를 유지한다.

> 💡 **왜 중요한가**: 데이터가 늘어날수록 OFFSET 페이지네이션은 선형적으로 느려지며, 커서 방식으로 전환하면 데이터 규모에 무관한 일정한 조회 성능을 확보할 수 있다.

### 2. PostgreSQL JSONB 컬럼과 GIN 인덱스 — 스키마 유연성과 검색 성능의 균형

JSONB 타입은 정형화되지 않은 속성(사용자 설정, 이벤트 메타데이터 등)을 별도 테이블 없이 하나의 컬럼에 저장할 수 있게 해준다. 단순 저장만 하면 전체 스캔이 필요하지만, GIN(Generalized Inverted Index) 인덱스를 걸면 `@>`, `?`, `?|` 같은 containment/existence 연산자를 활용한 빠른 검색이 가능하다. 다만 GIN 인덱스는 쓰기 시 오버헤드가 크므로, 읽기 비중이 높은 컬럼에 선택적으로 적용하는 것이 핵심이다.

> 💡 **왜 중요한가**: 모든 속성을 정규화하면 스키마 변경 비용이 커지고, JSONB를 무분별하게 쓰면 검색 성능이 무너지므로, GIN 인덱스와 함께 전략적으로 활용하는 것이 유연성과 성능을 동시에 잡는 방법이다.
