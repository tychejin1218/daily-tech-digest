### 1. `@Cacheable`과 Spring Cache 추상화로 반복 연산 성능 최적화

`@Cacheable`을 메서드에 선언하면 동일한 인자로 재호출 시 실제 로직을 실행하지 않고 캐시된 결과를 반환한다. `@CacheEvict`로 데이터 변경 시 캐시를 무효화하고, `@CachePut`으로 항상 실행하면서 결과를 갱신할 수 있다. 백엔드는 ConcurrentHashMap(기본), Caffeine, Redis 등으로 교체해도 애플리케이션 코드를 수정할 필요가 없다.

```java
@Service
public class ProductService {

    @Cacheable(value = "products", key = "#id", unless = "#result == null")
    public Product findById(Long id) {
        return productRepository.findById(id).orElse(null);
    }

    @CacheEvict(value = "products", key = "#product.id")
    public Product update(Product product) {
        return productRepository.save(product);
    }
}
```

```yaml
# application.yml (Caffeine 예시)
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=500,expireAfterWrite=10m
```

> 💡 **왜 중요한가**: DB/외부 API 호출이 잦은 조회 로직에 선언 한 줄로 응답 속도를 수십 배 줄일 수 있으며, 추상화 덕분에 캐시 구현체를 운영 환경에 맞게 교체해도 비즈니스 코드가 오염되지 않는다.

### 2. `ApplicationEventPublisher`로 이벤트 기반 아키텍처 구현해 서비스 간 결합도 낮추기

회원 가입 후 이메일 발송·포인트 적립 등 부가 작업을 직접 호출하면 서비스 간 강결합이 생긴다. `ApplicationEventPublisher`로 도메인 이벤트를 발행하고 `@EventListener`(또는 `@TransactionalEventListener`)로 처리하면 발행자와 구독자를 완전히 분리할 수 있다. `@TransactionalEventListener(phase = AFTER_COMMIT)`을 사용하면 트랜잭션이 커밋된 뒤에만 이벤트가 처리되어 데이터 정합성 문제도 방지된다.

```java
// 이벤트 클래스
public record UserRegisteredEvent(Long userId, String email) {}

// 발행 (UserService)
@Service
@RequiredArgsConstructor
public class UserService {
    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public User register(SignUpRequest request) {
        User user = userRepository.save(User.from(request));
        eventPublisher.publishEvent(new UserRegisteredEvent(user.getId(), user.getEmail()));
        return user;
    }
}

// 구독 (EmailService, PointService 각각 독립 선언)
@Component
public class EmailEventHandler {

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void sendWelcomeEmail(UserRegisteredEvent event) {
        // 이메일 발송 로직
    }
}
```

> 💡 **왜 중요한가**: 핵심 비즈니스 로직과 부가 작업을 이벤트로 분리하면 새 기능 추가 시 기존 코드를 수정하지 않아도 되어 OCP(개방-폐쇄 원칙)를 자연스럽게 지킬 수 있다.
