### 1. `ObjectProvider<T>`로 지연·선택적 의존성 주입 구현

Spring의 `ObjectProvider<T>`는 빈이 존재하지 않거나 여러 개일 때 안전하게 주입받을 수 있는 DI 전략입니다. `getIfAvailable()`로 빈 부재 시 기본값을 제공하고, `orderedStream()`으로 동일 타입의 다중 빈을 우선순위에 따라 스트림으로 처리할 수 있습니다. 직접 주입과 달리 빈 조회를 호출 시점까지 지연(lazy)시키므로, 순환 참조 문제를 우회하거나 선택적 확장 포인트를 설계할 때 특히 유용합니다.

```java
@Service
public class NotificationService {

    private final NotificationSender sender;

    public NotificationService(ObjectProvider<NotificationSender> senderProvider) {
        // 빈이 없으면 기본 구현체 사용
        this.sender = senderProvider.getIfAvailable(NoOpNotificationSender::new);
    }
}
```

> 💡 **왜 중요한가**: 플러그인 구조나 선택적 기능 모듈에서 빈 부재로 인한 `NoSuchBeanDefinitionException` 없이 유연한 의존성 해결이 가능해집니다.

### 2. Spring Boot 3.2+ Virtual Threads 활용으로 동시성 성능 극대화

Spring Boot 3.2부터 `spring.threads.virtual.enabled=true` 한 줄로 Tomcat, Jetty 등의 요청 처리에 Java 21 Virtual Threads를 적용할 수 있습니다. 기존 플랫폼 스레드 풀 기반에서는 I/O 대기 시 스레드가 블로킹되어 동시 처리량에 한계가 있었지만, Virtual Threads는 블로킹 호출 시 자동으로 캐리어 스레드를 반환하므로 적은 OS 자원으로 수천~수만 개의 동시 요청을 처리할 수 있습니다. 기존 동기 코드 스타일을 유지하면서도 WebFlux 수준의 동시성을 확보할 수 있어, 리액티브 전환 없이 성능을 개선하려는 프로젝트에 적합합니다.

```properties
# application.properties
spring.threads.virtual.enabled=true
```

```java
// 기존 동기 코드가 그대로 Virtual Thread 위에서 실행됨
@RestController
public class OrderController {

    @GetMapping("/orders/{id}")
    public OrderResponse getOrder(@PathVariable Long id) {
        // DB 조회 + 외부 API 호출이 블로킹되어도
        // Virtual Thread가 캐리어 스레드를 점유하지 않음
        Order order = orderService.findById(id);
        DeliveryStatus status = deliveryClient.getStatus(order.getTrackingId());
        return OrderResponse.of(order, status);
    }
}
```

> 💡 **왜 중요한가**: 리액티브 프로그래밍의 학습 비용과 코드 복잡성 없이, 설정 한 줄로 I/O 바운드 애플리케이션의 처리량을 비약적으로 향상시킬 수 있습니다.
