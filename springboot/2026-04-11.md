### 1. Testcontainers로 실제 DB 환경에서 통합 테스트 작성

`@DataJpaTest`의 H2 인메모리 DB는 실제 운영 DB(PostgreSQL, MySQL 등)와 방언(dialect) 차이로 인해 미묘한 버그를 놓치기 쉽습니다. Testcontainers를 사용하면 도커 컨테이너로 실제 DB를 테스트 시에 자동으로 띄우고 종료할 수 있어, H2 호환성 이슈 없이 운영 환경과 동일한 조건에서 JPA 레이어를 검증할 수 있습니다.

```kotlin
// build.gradle.kts
testImplementation("org.springframework.boot:spring-boot-testcontainers")
testImplementation("org.testcontainers:postgresql")
```

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
class OrderRepositoryTest {

    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16-alpine");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    OrderRepository orderRepository;

    @Test
    void 실제_PostgreSQL에서_주문_저장_조회() {
        Order order = new Order("user-1", BigDecimal.valueOf(15000));
        orderRepository.save(order);

        assertThat(orderRepository.findByUserId("user-1")).hasSize(1);
    }
}
```

`@DynamicPropertySource`로 컨테이너가 할당한 동적 포트를 스프링 컨텍스트에 주입하고, `Replace.NONE`으로 자동 교체를 막는 것이 핵심입니다. 컨테이너는 클래스 단위로 공유(`static`)해 시작 비용을 최소화하세요.

> 💡 **왜 중요한가**: H2로 통과한 테스트가 PostgreSQL의 JSON 타입이나 `RETURNING` 구문에서 깨지는 문제를 배포 전에 잡을 수 있다.

### 2. Spring Boot Actuator + Micrometer로 커스텀 비즈니스 메트릭 수집

Actuator는 JVM, HTTP 요청 수 같은 인프라 메트릭을 기본 제공하지만, "하루 결제 완료 건수"나 "쿠폰 발급 실패율" 같은 **비즈니스 메트릭**은 직접 등록해야 합니다. Micrometer의 `MeterRegistry`를 주입받아 `Counter`, `Gauge`, `Timer`를 선언하면 Prometheus/Grafana 등 기존 모니터링 스택과 즉시 연동됩니다.

```java
@Service
@RequiredArgsConstructor
public class PaymentService {

    private final MeterRegistry meterRegistry;
    private final Counter paymentSuccessCounter;
    private final Counter paymentFailCounter;

    @Bean
    public Counter paymentSuccessCounter(MeterRegistry registry) {
        return Counter.builder("payment.success")
            .description("결제 성공 건수")
            .tag("env", "prod")
            .register(registry);
    }

    @Bean
    public Counter paymentFailCounter(MeterRegistry registry) {
        return Counter.builder("payment.fail")
            .description("결제 실패 건수")
            .register(registry);
    }

    public PaymentResult pay(PaymentRequest request) {
        try {
            PaymentResult result = processPayment(request);
            paymentSuccessCounter.increment();
            return result;
        } catch (PaymentException e) {
            paymentFailCounter.increment();
            throw e;
        }
    }
}
```

```yaml
# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health, metrics, prometheus
  metrics:
    tags:
      application: ${spring.application.name}   # 모든 메트릭에 앱 이름 태그 자동 부착
```

`/actuator/metrics/payment.success`로 즉시 확인할 수 있고, `spring-boot-starter-actuator` + `micrometer-registry-prometheus` 의존성 추가만으로 Prometheus가 `/actuator/prometheus` 엔드포인트를 스크랩합니다. `Timer.record()`를 활용하면 특정 로직의 실행 시간 분포(p99 등)도 동일한 방식으로 수집할 수 있습니다.

> 💡 **왜 중요한가**: 인프라 지표만으로는 "서비스가 느린지" 알 수 있어도 "어떤 비즈니스 흐름이 문제인지"는 알 수 없으며, 커스텀 메트릭이 그 공백을 메운다.
