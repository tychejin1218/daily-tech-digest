### WAL(Write-Ahead Logging) — 데이터베이스가 충돌에도 데이터를 잃지 않는 이유

데이터베이스는 실제 데이터 파일을 디스크에 쓰기 **전에**, 변경 내용을 먼저 WAL(로그 파일)에 기록한다. 서버가 갑자기 죽어도 재시작 시 WAL을 재생(replay)해 커밋된 트랜잭션은 복구하고, 미완료 트랜잭션은 롤백한다. PostgreSQL의 `pg_wal`, MySQL InnoDB의 `redo log`가 모두 이 원리로 동작하며, `fsync` 설정을 끄면 성능은 올라가지만 전원 장애 시 데이터 손실 위험이 생긴다.

```sql
-- PostgreSQL: WAL 관련 설정 확인
SHOW wal_level;           -- minimal / replica / logical
SHOW synchronous_commit;  -- on(안전) / off(빠름, 손실 가능)

-- WAL 위치 조회 (복제 지연 모니터링에 활용)
SELECT pg_current_wal_lsn();
```

> 💡 **왜 중요한가**: `synchronous_commit = off` 같은 성능 최적화 옵션이 왜 위험한지, 그리고 Replica 복제 지연이 왜 발생하는지를 이해하려면 WAL 메커니즘을 알아야 한다.

### 윈도우 함수(Window Function) — GROUP BY 없이 집계하는 SQL의 핵심

`GROUP BY`는 행을 **묶어버려** 원본 데이터가 사라지지만, 윈도우 함수(`OVER`)는 원본 행을 유지하면서 집계 결과를 **옆에 붙여준다**. 누적 합계, 순위, 이동 평균처럼 "각 행과 그룹 집계를 동시에 보고 싶을 때" 서브쿼리 없이 한 번의 스캔으로 처리할 수 있어 성능과 가독성이 모두 올라간다.

```sql
-- 각 주문 행을 유지하면서 고객별 누적 매출 계산
SELECT
    order_id,
    customer_id,
    amount,
    SUM(amount)  OVER (PARTITION BY customer_id ORDER BY order_date) AS cumulative_sales,
    RANK()       OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rank_in_customer,
    LAG(amount)  OVER (PARTITION BY customer_id ORDER BY order_date) AS prev_amount
FROM orders;

-- JPA에서는 네이티브 쿼리 또는 Blaze-Persistence로 사용
@Query(value = """
    SELECT *, RANK() OVER (PARTITION BY customer_id ORDER BY amount DESC)
    FROM orders
    """, nativeQuery = true)
List<Object[]> findWithRank();
```

> 💡 **왜 중요한가**: 순위·누적·이동 평균을 구하기 위해 작성하던 복잡한 자기 조인이나 서브쿼리를 윈도우 함수 한 줄로 대체하면 실행 계획이 단순해지고 쿼리 성능이 크게 개선된다.
