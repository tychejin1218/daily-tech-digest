### 1. Scoped Values로 스레드 간 불변 데이터 안전하게 공유하기

`ThreadLocal`은 가변(mutable) 상태를 스레드별로 격리하지만, Virtual Thread 환경에서는 수천 개의 인스턴스가 생성되어 메모리 누수와 예측 불가한 상태 공유 문제가 생긴다. Java 21에서 도입된 `ScopedValue`는 불변 값을 특정 실행 범위(scope) 내에서만 읽을 수 있도록 제한하며, 명시적인 `remove()` 없이도 범위 이탈 시 자동 해제된다.

```java
// ScopedValue 선언 (불변, 재할당 불가)
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

// 요청 처리 시 범위 지정
ScopedValue.where(CURRENT_USER, user).run(() -> {
    processRequest();     // 이 범위 안에서만 CURRENT_USER 접근 가능
    callDownstream();
});

// 어디서든 꺼내 쓰기 — 범위 밖에서 호출하면 NoSuchElementException
User u = CURRENT_USER.get();
```

`ThreadLocal`과 달리 값 변경이 불가능하므로 동시성 버그의 원인이 되는 공유 가변 상태를 원천 차단한다. `run()` 블록이 끝나면 값은 자동으로 사라진다.

> 💡 **왜 중요한가**: Virtual Thread와 함께 사용할 때 `ThreadLocal` 대비 메모리 효율이 크게 향상되며, 요청 컨텍스트(인증 정보, 트레이스 ID 등)를 전파하는 패턴의 사실상 표준이 된다.

### 2. `instanceof` 패턴 매칭으로 타입 검사와 캐스팅 한 번에 처리하기

Java 16에서 정식화된 `instanceof` 패턴 매칭은 타입 확인과 동시에 바인딩 변수를 선언해, 반복적인 캐스팅 코드를 제거한다. 조건 표현식 내에서 바로 바인딩된 변수를 사용할 수 있어 가독성이 높아진다.

```java
// 기존 방식
if (obj instanceof String) {
    String s = (String) obj;   // 중복 캐스팅
    System.out.println(s.toUpperCase());
}

// 패턴 매칭
if (obj instanceof String s) {
    System.out.println(s.toUpperCase());   // s는 이미 String으로 바인딩
}

// 조건과 결합도 가능
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s.substring(0, 5));
}

// equals() 구현 패턴
@Override
public boolean equals(Object o) {
    return o instanceof Point p
        && p.x == this.x
        && p.y == this.y;
}
```

`switch` 패턴 매칭(Java 21)의 기반이 되는 기능이며, `Sealed Class`와 결합하면 컴파일러가 케이스 누락 여부를 정적으로 검증해준다.

> 💡 **왜 중요한가**: 불필요한 캐스팅 제거로 코드가 짧아지는 것을 넘어, `ClassCastException` 가능성을 구조적으로 없애고 컴파일러가 타입 안전성을 보장하게 된다.
