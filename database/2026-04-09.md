### Connection Pool 튜닝 — HikariCP `maximumPoolSize` 오해와 실전 설정

애플리케이션 서버의 CPU 코어 수보다 `maximumPoolSize`를 무조건 크게 잡으면 오히려 Context Switching 오버헤드와 DB 서버 부하가 증가한다. HikariCP 공식 권고는 **`(CPU 코어 수 × 2) + 유효 스핀들 수`**를 기준으로 삼되, 실제 TPS와 쿼리 평균 응답 시간으로 `커넥션 수 = TPS × 평균응답시간(초)` 공식(Little's Law)을 검증하는 것이다. `connectionTimeout`(커넥션 대기 한계)과 `idleTimeout`(유휴 커넥션 회수)도 함께 조정해 Pool 고갈과 불필요한 커넥션 유지를 동시에 방지해야 한다.

```yaml
# Spring Boot application.yml 예시
spring:
  datasource:
    hikari:
      maximum-pool-size: 10        # (코어 수 × 2) + 1 기준으로 시작
      minimum-idle: 5
      connection-timeout: 3000     # 3초 초과 시 예외 → 빠른 장애 감지
      idle-timeout: 600000         # 10분 유휴 시 반환
      max-lifetime: 1800000        # 30분 → DB wait_timeout보다 짧게
```

> 💡 **왜 중요한가**: Pool 사이즈 과대 설정은 "커넥션이 많으면 빠르다"는 직관과 반대로 DB 스레드 경합을 늘려 전체 처리량을 낮추며, 실제 장애 원인 1순위 중 하나다.

### MVCC(Multi-Version Concurrency Control) — 읽기가 쓰기를 막지 않는 이유

PostgreSQL·MySQL InnoDB는 데이터를 수정할 때 기존 행을 덮어쓰지 않고 **새 버전을 추가**하며, 각 트랜잭션은 자신의 시작 시점 스냅샷 기준의 버전을 읽는다. 덕분에 `SELECT`는 `UPDATE`/`DELETE`와 충돌 없이 동시에 실행되고 락 경합이 최소화된다. 단, 오래된 버전 데이터(Dead Tuple / Undo Log)는 주기적으로 정리되지 않으면 테이블 비대화(Table Bloat)와 성능 저하를 일으키므로, PostgreSQL은 `VACUUM`, MySQL은 Purge Thread 동작을 모니터링해야 한다.

```sql
-- PostgreSQL: Dead Tuple 현황 확인
SELECT relname, n_live_tup, n_dead_tup,
       round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 2) AS dead_ratio
FROM   pg_stat_user_tables
ORDER  BY n_dead_tup DESC;

-- dead_ratio가 10~20% 이상이면 VACUUM ANALYZE 또는 autovacuum 파라미터 점검
```

> 💡 **왜 중요한가**: MVCC 내부 동작을 모르면 "락도 없는데 왜 쿼리가 느리지?"라는 Table Bloat 문제를 트랜잭션 격리 이슈로 오진해 해결에 오랜 시간을 낭비하게 된다.
