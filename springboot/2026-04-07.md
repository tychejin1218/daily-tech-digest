# Spring Boot - 2026-04-07

### 1. `@Transactional` 전파 속성(Propagation)의 함정

`@Transactional`은 기본적으로 `REQUIRED` 전파 속성을 사용하여 기존 트랜잭션에 참여하지만, **같은 클래스 내부에서 호출하면 프록시를 거치지 않아 트랜잭션이 적용되지 않는다.** 이를 Self-Invocation 문제라 하며, 해결하려면 해당 메서드를 별도 빈으로 분리하거나 `ApplicationContext`를 통해 자기 자신의 프록시를 주입받아야 한다.

```java
// 잘못된 패턴 - 트랜잭션 미적용
public void outer() {
    this.inner(); // 프록시 우회
}

// 올바른 패턴 - 별도 서비스로 분리
public void outer() {
    innerService.inner(); // 프록시 통과
}
```

> 💡 **왜 중요한가**: Self-Invocation 버그는 테스트 환경에서는 잘 드러나지 않다가 운영 중 데이터 정합성 문제로 이어질 수 있다.

### 2. Spring Security의 `SecurityFilterChain` 다중 설정으로 API/Web 분리

Spring Security 6.x부터 `WebSecurityConfigurerAdapter`가 제거되고 `SecurityFilterChain` 빈 방식으로 전환되었다. `@Order` 어노테이션을 활용해 `/api/**` 경로는 JWT 기반 Stateless 정책을, 나머지 경로는 세션 기반 정책을 적용하는 **다중 필터 체인**을 선언적으로 구성할 수 있다.

```java
@Bean
@Order(1)
public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception {
    return http
        .securityMatcher("/api/**")
        .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
        .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
}

@Bean
@Order(2)
public SecurityFilterChain webFilterChain(HttpSecurity http) throws Exception {
    return http
        .formLogin(Customizer.withDefaults())
        .build();
}
```

> 💡 **왜 중요한가**: 하나의 설정으로 REST API와 웹 UI를 동시에 지원하면 보안 정책이 충돌하므로, 명시적으로 분리하는 것이 유지보수와 보안 모두에 유리하다.
