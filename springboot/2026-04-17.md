### 1. `@ControllerAdvice`와 `@ExceptionHandler`로 전역 예외 처리 전략 구축

컨트롤러마다 try-catch를 반복하는 대신, `@ControllerAdvice` 클래스에 `@ExceptionHandler` 메서드를 집중시키면 애플리케이션 전체의 예외를 일관된 응답 형식으로 변환할 수 있습니다. 비즈니스 예외, 검증 예외, 예상치 못한 시스템 예외를 계층별로 분리해 처리하면 클라이언트는 항상 예측 가능한 에러 응답을 받게 됩니다. `ResponseEntityExceptionHandler`를 상속하면 Spring MVC가 던지는 표준 예외(MethodArgumentNotValidException, HttpRequestMethodNotSupportedException 등)까지 한 곳에서 커스터마이징할 수 있습니다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusiness(BusinessException ex) {
        return ResponseEntity
            .status(ex.getStatus())
            .body(new ErrorResponse(ex.getCode(), ex.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleUnexpected(Exception ex) {
        log.error("Unhandled exception", ex);
        return ResponseEntity
            .internalServerError()
            .body(new ErrorResponse("INTERNAL_ERROR", "서버 내부 오류가 발생했습니다."));
    }
}
```

> 💡 **왜 중요한가**: 예외 처리 로직을 한 곳으로 모으면 컨트롤러는 비즈니스 로직에만 집중하고, API 에러 응답의 일관성과 유지보수성이 크게 향상됩니다.

### 2. Spring Data JPA `@EntityGraph`로 N+1 쿼리 문제 해결

지연 로딩(Lazy Loading) 설정된 연관 엔티티를 반복 접근하면 N+1 쿼리가 발생해 성능이 급격히 저하됩니다. `@EntityGraph`를 리포지토리 메서드에 선언하면 JPQL을 직접 작성하지 않고도 특정 조회 시점에만 페치 조인을 적용할 수 있어, 엔티티 기본 매핑은 Lazy로 유지하면서 필요한 곳에서만 Eager 로딩을 선택적으로 수행할 수 있습니다. `attributePaths`에 중첩 경로(`"order.items"`)도 지정 가능하므로 다단계 연관관계도 한 번의 쿼리로 해결됩니다.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {

    // 기본 조회 — Team은 Lazy 로딩
    Optional<Member> findById(Long id);

    // Team까지 한 번에 페치 조인
    @EntityGraph(attributePaths = {"team", "team.department"})
    @Query("SELECT m FROM Member m WHERE m.active = true")
    List<Member> findAllActiveWithTeam();
}
```

> 💡 **왜 중요한가**: N+1 문제는 JPA 기반 애플리케이션에서 가장 흔한 성능 병목이며, `@EntityGraph`를 활용하면 엔티티 설계를 변경하지 않고 쿼리 단위로 페치 전략을 제어할 수 있습니다.
