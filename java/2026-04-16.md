### 1. Record Patterns (레코드 패턴)으로 중첩 데이터 구조 분해하기 (Java 21 정식)

Java 21에서 정식 도입된 Record Patterns(JEP 440)는 `instanceof`나 `switch`에서 레코드의 컴포넌트를 **한 번에 구조 분해(destructuring)** 할 수 있게 해줍니다. 중첩 레코드도 한 표현식에서 깊이 있게 분해할 수 있어, 기존에 `getter`를 연쇄 호출하던 코드를 크게 줄여줍니다.

```java
record Point(int x, int y) {}
record Line(Point start, Point end) {}

// 기존 방식: 단계별로 꺼내야 함
if (obj instanceof Line line) {
    int x1 = line.start().x();
    int y1 = line.start().y();
    // ...
}

// Record Pattern: 중첩 구조를 한 번에 분해
if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    double distance = Math.sqrt(Math.pow(x2 - x1, 2) + Math.pow(y2 - y1, 2));
}

// switch와 결합하면 더 강력
static String describe(Object obj) {
    return switch (obj) {
        case Line(Point(var x, var y), Point(var x2, var y2))
            when x == x2 -> "수직선";
        case Line(Point(var x, var y), Point(var x2, var y2))
            when y == y2 -> "수평선";
        case Line l -> "일반 직선";
        default -> "직선 아님";
    };
}
```

> 💡 **왜 중요한가**: Sealed Classes + Record + Record Patterns + switch 패턴 매칭을 조합하면 Java에서도 함수형 언어 수준의 **대수적 데이터 타입(ADT) 패턴**을 구현할 수 있어, 복잡한 도메인 모델을 타입 안전하고 간결하게 다룰 수 있습니다.

### 2. Generational ZGC로 초저지연 가비지 컬렉션 달성하기 (Java 21 정식)

Java 21에서 Generational ZGC(JEP 439)가 정식 도입되면서, ZGC가 **세대별 수집(generational collection)** 을 지원하게 되었습니다. 대부분의 객체는 생성 직후 곧 소멸된다는 "약한 세대 가설"을 활용해, Young 세대를 더 자주·빠르게 수집하고 Old 세대는 필요할 때만 처리합니다. 이를 통해 기존 비세대 ZGC 대비 **처리량(throughput)은 향상**되면서도 **일시 정지 시간은 1ms 이하**를 유지합니다.

```bash
# Java 21: Generational ZGC 활성화 (기본값은 아직 non-generational)
java -XX:+UseZGC -XX:+ZGenerational -Xmx4g -jar app.jar

# Java 23+: Generational ZGC가 기본값으로 변경됨
# -XX:+ZGenerational 플래그 불필요
java -XX:+UseZGC -Xmx4g -jar app.jar

# GC 로그로 세대별 수집 동작 확인
java -XX:+UseZGC -XX:+ZGenerational -Xlog:gc*:file=gc.log -jar app.jar
```

```
# GC 로그 예시 — Young/Old 세대 구분 확인
[gc,start] GC(42) Minor Collection (Young Generation)
[gc      ] GC(42) Minor Collection 512M->48M (1024M) 0.823ms
[gc,start] GC(43) Major Collection (Old Generation)  
[gc      ] GC(43) Major Collection 896M->320M (2048M) 1.102ms
```

> 💡 **왜 중요한가**: 수백 GB 힙에서도 1ms 이하의 정지 시간을 보장하므로, 금융 트레이딩·실시간 게임 서버·대규모 캐시 시스템 등 **지연 시간에 민감한 Java 애플리케이션**에서 G1GC를 대체할 수 있는 실전적인 선택지입니다.
