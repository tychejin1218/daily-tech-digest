### 낙관적 락(Optimistic Lock) vs 비관적 락(Pessimistic Lock)

낙관적 락은 충돌이 드물다고 가정하고 실제 커밋 시점에 버전 번호(`@Version`)를 비교해 충돌을 감지하며, 비관적 락은 조회 시점에 `SELECT ... FOR UPDATE`로 행을 즉시 잠근다. 낙관적 락은 읽기 비중이 높은 서비스에서 DB 잠금 없이 처리량을 높여주지만, 충돌 시 `OptimisticLockException`을 애플리케이션에서 직접 재시도 처리해야 한다. 비관적 락은 충돌이 잦거나 정합성이 절대적인 재고 차감·포인트 사용 같은 도메인에 적합하나, 잠금 대기로 인한 데드락 위험이 있다.

```java
// 낙관적 락: @Version 필드로 자동 관리
@Entity
public class Product {
    @Id Long id;
    int stock;

    @Version  // UPDATE 시 WHERE version = :old 조건 자동 추가
    Long version;
}

// 비관적 락: 조회 시점에 행 잠금
public interface ProductRepository extends JpaRepository<Product, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdForUpdate(@Param("id") Long id);
}
```

> 💡 **왜 중요한가**: 동시성 전략을 잘못 선택하면 재고 음수, 포인트 중복 사용 같은 데이터 무결성 버그가 운영 중 발생하고 재현이 어렵다.

### Slow Query 로그 분석과 최적화 워크플로

MySQL의 `slow_query_log`를 활성화하면 `long_query_time`을 초과하는 쿼리를 자동으로 기록하며, `mysqldumpslow`나 `pt-query-digest`로 집계해 가장 빈번하거나 총 실행 시간이 긴 쿼리를 우선순위화할 수 있다. 포착된 쿼리는 `EXPLAIN` → 인덱스 추가 또는 쿼리 재작성 → 재측정의 사이클로 개선한다. `log_queries_not_using_indexes = ON`을 함께 설정하면 풀스캔 쿼리도 별도 캡처할 수 있다.

```sql
-- slow query log 활성화 (my.cnf 또는 동적 설정)
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;          -- 1초 초과 쿼리 기록
SET GLOBAL log_queries_not_using_indexes = ON;

-- 로그 파일 위치 확인
SHOW VARIABLES LIKE 'slow_query_log_file';
```

```bash
# 실행 횟수·평균 시간 기준으로 상위 10개 쿼리 집계
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# pt-query-digest (더 상세한 분석)
pt-query-digest /var/log/mysql/slow.log | head -100
```

> 💡 **왜 중요한가**: 성능 문제는 전체 쿼리 중 소수의 쿼리가 대부분의 DB 부하를 유발하므로, 슬로우 쿼리 로그로 병목을 정량적으로 찾는 것이 최적화의 첫 단계다.
