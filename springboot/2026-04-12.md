### HikariCP 커넥션 풀 튜닝으로 DB 병목 제거

Spring Boot의 기본 커넥션 풀인 HikariCP는 기본값 그대로 두면 트래픽이 몰릴 때 `Connection is not available` 타임아웃이 발생하기 쉽다. `maximum-pool-size`는 CPU 코어 수 × 2 + 유효 디스크 수 공식을 기준으로 잡고, `connection-timeout`과 `idle-timeout`을 서비스 특성에 맞게 조정해야 한다.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10          # 동시 DB 연결 최대 수
      minimum-idle: 5                # 유휴 상태에서 유지할 최소 연결 수
      connection-timeout: 3000       # 연결 획득 대기 최대 시간 (ms)
      idle-timeout: 600000           # 유휴 연결 유지 시간 (ms)
      max-lifetime: 1800000          # 연결 최대 수명 (ms, DB 타임아웃보다 짧게)
      pool-name: MainHikariPool
```

풀 크기를 무조건 크게 잡으면 오히려 DB 서버에서 컨텍스트 스위칭 비용이 올라간다. `spring.datasource.hikari.register-mbeans=true`로 JMX를 활성화하면 Actuator와 함께 실시간 풀 상태(`active`, `idle`, `pending` 연결 수)를 모니터링할 수 있다.

> 💡 **왜 중요한가**: 커넥션 풀 설정은 애플리케이션 코드 변경 없이 처리량과 응답 시간에 가장 즉각적인 영향을 미치는 인프라 레벨 튜닝 포인트다.

### Resilience4j Circuit Breaker로 외부 서비스 장애 격리

외부 API 호출이 연속 실패하면 스레드 풀이 고갈되어 전체 서비스가 멈출 수 있다. `spring-cloud-starter-circuitbreaker-resilience4j`를 사용하면 실패율이 임계치를 넘는 순간 회로를 열어 즉시 폴백을 반환하고, 일정 시간 후 `HALF_OPEN` 상태로 자동 복구를 시도한다.

```java
// application.yml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        failure-rate-threshold: 50        # 실패율 50% 초과 시 OPEN
        slow-call-rate-threshold: 80      # 느린 호출 80% 초과 시 OPEN
        slow-call-duration-threshold: 2s
        wait-duration-in-open-state: 30s  # OPEN → HALF_OPEN 대기 시간
        permitted-calls-in-half-open-state: 3

// Service
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
public PaymentResult charge(PaymentRequest request) {
    return paymentClient.charge(request);
}

private PaymentResult paymentFallback(PaymentRequest request, Exception e) {
    log.warn("Payment service unavailable, queuing for retry", e);
    return PaymentResult.queued(request.getOrderId());
}
```

`@Retry`와 함께 사용할 때는 `@Retry` → `@CircuitBreaker` 순으로 중첩해야 재시도 후 실패가 회로 차단 카운트에 올바르게 집계된다.

> 💡 **왜 중요한가**: 장애 격리 없이 외부 의존성 하나의 지연이 전체 서비스의 스레드 풀을 잠식하는 cascading failure는 MSA 환경에서 가장 흔한 장애 패턴이다.
