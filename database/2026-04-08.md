### 트랜잭션 격리 수준(Isolation Level)과 발생 가능한 이상 현상

트랜잭션 격리 수준은 `READ UNCOMMITTED` → `READ COMMITTED` → `REPEATABLE READ` → `SERIALIZABLE` 순으로 높아지며, 격리 수준이 높을수록 동시성은 낮아지고 정합성은 높아진다. 각 수준은 Dirty Read, Non-Repeatable Read, Phantom Read 세 가지 이상 현상을 방지하는 범위가 다르다. MySQL InnoDB의 기본값은 `REPEATABLE READ`이며, 이 수준에서는 MVCC(Multi-Version Concurrency Control)를 통해 Phantom Read도 대부분 방지된다.

```sql
-- 세션별 격리 수준 설정
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 현재 격리 수준 확인
SELECT @@transaction_isolation;
```

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 방지 | 발생 | 발생 |
| REPEATABLE READ | 방지 | 방지 | 발생* |
| SERIALIZABLE | 방지 | 방지 | 방지 |

> 💡 **왜 중요한가**: 격리 수준을 모르면 금융·재고 같은 정합성이 중요한 도메인에서 데이터 불일치 버그가 발생해도 원인을 찾기 어렵다.

### 복합 인덱스(Composite Index) 컬럼 순서 설계 전략

복합 인덱스는 **카디널리티가 높은 컬럼을 앞에**, **등호(`=`) 조건 컬럼을 범위(`<`, `>`, `BETWEEN`) 조건 컬럼보다 앞에** 배치해야 인덱스 효율이 극대화된다. B-Tree 인덱스는 선두 컬럼부터 순서대로 정렬되므로, `WHERE` 절에서 선두 컬럼을 건너뛰면 인덱스 전체가 사용되지 않는 **인덱스 스킵** 문제가 발생한다. 예를 들어 `(status, created_at)` 인덱스는 `WHERE status = 'ACTIVE' AND created_at > '2026-01-01'` 쿼리에 유효하지만, `WHERE created_at > '2026-01-01'`만 단독으로 사용하면 해당 인덱스를 활용하지 못한다.

```sql
-- 좋은 예: 등호 조건(status)을 앞에, 범위 조건(created_at)을 뒤에
CREATE INDEX idx_order_status_date ON orders(status, created_at);

-- EXPLAIN으로 인덱스 사용 여부 확인
EXPLAIN SELECT * FROM orders
WHERE status = 'ACTIVE' AND created_at > '2026-01-01';
-- → type: range, key: idx_order_status_date (효율적)

EXPLAIN SELECT * FROM orders
WHERE created_at > '2026-01-01';
-- → type: ALL (풀 스캔, 인덱스 미사용)
```

> 💡 **왜 중요한가**: 인덱스를 만들었는데도 쿼리가 느린 이유의 상당수는 컬럼 순서 설계 실수이므로, `EXPLAIN` 분석과 함께 순서 원칙을 이해하는 것이 필수다.
