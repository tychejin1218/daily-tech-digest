### 1. `@Validated` + Bean Validation으로 계층별 입력 검증 분리

`@RequestBody`에 `@Valid`만 달면 컨트롤러 계층에서만 검증이 끝나지만, 서비스 내부 메서드나 파라미터 단위 검증은 놓치기 쉽다. 클래스 레벨에 `@Validated`를 선언하면 Spring AOP가 프록시를 생성해 메서드 파라미터·반환값에 선언된 `@NotNull`, `@Size` 등 제약 어노테이션을 자동으로 실행한다. 검증 실패 시 `ConstraintViolationException`이 발생하므로 `@ExceptionHandler`에서 400 응답으로 통일하면 된다.

```kotlin
// build.gradle
implementation("org.springframework.boot:spring-boot-starter-validation")

// Service
@Service
@Validated                          // ← 클래스 레벨에 선언
class OrderService {

    fun createOrder(@Valid request: CreateOrderRequest): OrderResponse { ... }

    fun getOrder(@Positive orderId: Long): OrderResponse { ... }  // 파라미터 직접 검증
}

// DTO
data class CreateOrderRequest(
    @field:NotBlank val productCode: String,
    @field:Positive val quantity: Int,
    @field:DecimalMin("0.0") val discountRate: BigDecimal
)

// 예외 핸들러
@RestControllerAdvice
class ValidationHandler {

    @ExceptionHandler(ConstraintViolationException::class)
    fun handleConstraintViolation(e: ConstraintViolationException): ResponseEntity<Map<String, Any>> {
        val errors = e.constraintViolations.associate {
            it.propertyPath.toString() to it.message
        }
        return ResponseEntity.badRequest().body(mapOf("errors" to errors))
    }
}
```

> 💡 **왜 중요한가**: 컨트롤러 DTO 검증만으로는 서비스 내부 호출·배치 처리 경로를 통한 잘못된 데이터 유입을 막을 수 없으므로, 계층별 검증을 선언적으로 강제해 방어 깊이를 높인다.

### 2. `@Retryable`로 외부 API 재시도와 `@Recover`로 폴백 처리

네트워크 순단이나 외부 서비스의 일시적 오류(5xx)에 단순 예외 전파로 대응하면 사용자 경험이 크게 떨어진다. Spring Retry의 `@Retryable`을 붙이면 지정한 예외가 발생했을 때 백오프(backoff) 전략에 따라 자동 재시도하고, 재시도를 모두 소진하면 `@Recover` 메서드로 폴백 로직을 실행한다. `@EnableRetry`를 설정 클래스에 추가하는 것만으로 AOP 기반으로 동작하므로 기존 코드 변경이 최소화된다.

```kotlin
// build.gradle
implementation("org.springframework.retry:spring-retry")
implementation("org.springframework.boot:spring-boot-starter-aop") // AOP 필수

// 설정
@Configuration
@EnableRetry
class RetryConfig

// 서비스
@Service
class PaymentGatewayClient(private val restClient: RestClient) {

    @Retryable(
        retryFor = [RestClientException::class, HttpServerErrorException::class],
        maxAttempts = 3,
        backoff = Backoff(delay = 500, multiplier = 2.0)  // 500ms → 1s → 2s
    )
    fun charge(request: PaymentRequest): PaymentResponse {
        return restClient.post()
            .uri("/charge")
            .body(request)
            .retrieve()
            .body(PaymentResponse::class.java)!!
    }

    @Recover  // 시그니처: 반환 타입 동일, 첫 파라미터는 발생한 예외
    fun chargeRecover(e: Exception, request: PaymentRequest): PaymentResponse {
        log.error("결제 게이트웨이 재시도 소진: orderId=${request.orderId}", e)
        return PaymentResponse(status = "PENDING", message = "나중에 재처리 예정")
    }
}
```

> 💡 **왜 중요한가**: 외부 의존성의 일시적 장애를 코드 레벨 retry 로직 없이 선언적으로 흡수할 수 있어, 서비스 복원력(resilience)을 높이면서 비즈니스 로직을 오염시키지 않는다.
