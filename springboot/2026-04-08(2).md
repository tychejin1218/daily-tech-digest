### 1. Spring Data JPA N+1 문제와 `@EntityGraph`로 해결하기

연관 엔티티를 `LAZY` 로딩으로 설정했을 때, 컬렉션을 조회하면 각 요소마다 추가 쿼리가 발생하는 N+1 문제가 생긴다. `@EntityGraph`를 사용하면 JPQL을 직접 수정하지 않고도 특정 조회 메서드에만 즉시 로딩(Fetch Join)을 적용할 수 있다.

```java
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    private Member member;
}

// Repository
public interface OrderRepository extends JpaRepository<Order, Long> {

    // N+1 발생: Order 100건 조회 시 Member 쿼리 100번 추가
    List<Order> findAll();

    // EntityGraph로 해결: 단일 JOIN FETCH 쿼리 실행
    @EntityGraph(attributePaths = {"member"})
    List<Order> findAllWithMember();
}
```

`FETCH JOIN` JPQL보다 선언적이고, 필요한 메서드에만 선택적으로 적용할 수 있어 유연하다. 단, 컬렉션(`@OneToMany`) 다중 `@EntityGraph` 동시 적용은 `MultipleBagFetchException`을 유발하므로 하나씩 적용하거나 `Set` 타입을 사용해야 한다.

> 💡 **왜 중요한가**: N+1 문제는 개발 환경에서는 눈에 띄지 않다가 운영 데이터가 쌓이면 성능 장애의 주범이 되므로 반드시 쿼리 로그로 확인하고 잡아야 한다.

### 2. `@ControllerAdvice` + `@ExceptionHandler`로 전역 예외 처리 통합

각 컨트롤러마다 `try-catch`를 반복하는 대신 `@ControllerAdvice` 클래스 하나에서 애플리케이션 전체의 예외를 일관된 응답 형식으로 처리할 수 있다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(EntityNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse("NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("VALIDATION_FAILED", message));
    }

    @ExceptionHandler(Exception.class)  // 최후 방어선
    public ResponseEntity<ErrorResponse> handleAll(Exception e) {
        return ResponseEntity.internalServerError()
            .body(new ErrorResponse("INTERNAL_ERROR", "서버 오류가 발생했습니다."));
    }
}

record ErrorResponse(String code, String message) {}
```

`@RestControllerAdvice`는 `@ControllerAdvice` + `@ResponseBody`의 조합으로, REST API에서는 이쪽을 사용한다. 예외 계층 구조에서 더 구체적인 타입의 핸들러가 우선 적용되므로 `Exception.class` 핸들러는 항상 마지막에 정의한다.

> 💡 **왜 중요한가**: 예외 처리를 한 곳에서 관리하면 API 응답 형식이 일관되고, 보안상 민감한 스택 트레이스가 클라이언트에 노출되는 것을 방지할 수 있다.
